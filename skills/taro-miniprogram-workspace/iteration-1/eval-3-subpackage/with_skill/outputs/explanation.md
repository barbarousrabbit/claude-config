# 分包方案说明

## 问题背景

主包当前 2.3MB，超出微信小程序 2MB 限制，build 时出现警告。需要将页面合理分配到分包，使主包压缩到 2MB 以下。

## 核心原则

- **主包只保留 Tab Bar 页面**：小程序启动时必须下载主包，因此只把用户打开即需要的页面放进主包。
- **按用户操作链路分包**：将功能相关的页面归到同一分包，减少跨包跳转的概率，同时让每个分包的预加载目标更明确。
- **预加载利用空闲带宽**：在「上游页面」展示时触发下游分包的预加载，消除跳转时的白屏等待。

## 分包划分

| 分包 | root | 包含页面 | 进入入口 |
|------|------|----------|----------|
| 主包 | — | 首页、购物车、个人中心 | 启动 / Tab Bar |
| packageProduct | `packageProduct/` | 搜索、商品详情 | 首页 |
| packageOrder | `packageOrder/` | 订单列表、订单详情、结算 | 购物车 / 个人中心 |
| packageRefund | `packageRefund/` | 申请退款、退款详情、退货物流 | 订单详情 |
| packageService | `packageService/` | 客服聊天、地址列表、地址编辑、设置 | 个人中心 |

### 主包页面选择理由

只有三个 Tab Bar 页面（首页、购物车、个人中心）放入主包。这三个页面是用户打开小程序时微信强制加载的，无法按需加载；其余所有页面均属于用户主动操作后才会访问，适合放入分包。

### packageProduct - 商品分包

搜索和商品详情是首页的直接下游，用户浏览首页后极大概率会进入商品浏览链路。独立成分包后，可以在首页展示时进行预加载，几乎无感知。

### packageOrder - 订单分包

订单列表、订单详情、结算页构成完整的下单流程。结算页（checkout）从购物车直接跳入，和订单列表/详情归属同一用户旅程，放在一起合理。

### packageRefund - 售后分包

退款三个页面（申请、详情、物流）是低频操作，且只能从订单详情跳入。单独拆包不会影响主流程，还能为其他分包节省体积空间。

### packageService - 服务分包

客服聊天含 WebSocket 相关逻辑，代码体积较重；地址管理和设置页均从个人中心进入，归属同一分包便于统一预加载。

## 预加载策略

| 触发页面 | 预加载分包 | 网络限制 | 原因 |
|----------|-----------|----------|------|
| 首页 | packageProduct | all | 用户最可能浏览商品，全网络预加载保障体验 |
| 购物车 | packageOrder | all | 下单是购物车的核心出口，全网络预加载 |
| 个人中心 | packageOrder + packageService | wifi | 两个分包同时预加载，限 wifi 防流量消耗 |
| 订单详情 | packageRefund | wifi | 售后低频，仅 wifi 下预加载，不强占流量 |

## 验证方式

构建后查看各包体积：

```bash
pnpm --filter customer-app build:weapp
# 查看 dist/ 目录
# 主包：dist/pages/ + dist/app.js 等
# 分包：dist/packageProduct/ dist/packageOrder/ 等
```

若主包仍超限，优先排查：
1. 是否有大图片资源放在主包 `assets/` 下（应迁移至 CDN / 分包 assets）
2. 是否有只在分包页面使用的组件被打入主包（检查 import 路径）
3. Tab Bar 图标是否过大（建议每张 < 3KB）

# Taro 小程序分包方案说明

## 核心原则

微信小程序对主包有严格的 2MB 限制。主包体积超标的根本原因通常是将不属于启动必需的页面和资源放入主包。解决思路是：**主包只放 TabBar 页面，其余页面全部移入分包**。

---

## 分包划分方案

### 主包（`pages/`）— 目标 < 2MB

| 页面 | 路径 | 原因 |
|------|------|------|
| 首页 | `pages/index/index` | TabBar 页，必须在主包 |
| 分类 | `pages/category/index` | TabBar 页，必须在主包 |
| 购物车 | `pages/cart/index` | TabBar 页，必须在主包 |
| 个人中心 | `pages/profile/index` | TabBar 页，必须在主包 |

> 微信小程序规定：TabBar 中引用的页面必须在主包，不能放入分包。因此主包页面数量由 TabBar 数量决定，4 个 TabBar 页是不可减少的底线。

---

### 分包 1：商品分包（`packageProduct`）

| 页面 | 说明 |
|------|------|
| 商品详情 | 核心浏览页面，流量最高 |
| 搜索 | 独立功能，可懒加载 |
| 店铺主页 | 从商品详情跳转，低频 |

**划分理由**：商品浏览是独立的内容消费场景，与购物车/订单的业务逻辑分离。该分包体积会随商品详情的富文本渲染组件增大，单独隔离避免影响主包。

---

### 分包 2：订单分包（`packageOrder`）

| 页面 | 说明 |
|------|------|
| 结算 | 下单核心链路 |
| 订单列表 | 查看历史订单 |
| 订单详情 | 单笔订单操作入口 |
| 代付 | 转账确认页面 |

**划分理由**：下单-支付-查看订单是连续的操作链路，放在同一分包可以减少分包切换时的加载等待。结算页面通常包含地图/物流组件，体积较大，必须移出主包。

---

### 分包 3：售后分包（`packageAftersale`）

| 页面 | 说明 |
|------|------|
| 申请退款 | 退款发起页 |
| 退款详情 | 退款状态追踪 |
| 退货物流 | 填写寄回信息 |
| 客服聊天 | WebSocket 长连接，资源占用大 |
| 消息列表 | 消息中心入口 |

**划分理由**：售后场景使用频率低（相对于浏览和购买），且客服聊天模块通常引入 WebSocket、富文本编辑器等重型依赖，必须独立分包，避免污染高频路径的加载性能。

---

### 分包 4：设置分包（`packageSettings`）

| 页面 | 说明 |
|------|------|
| 地址列表 | 收货地址管理 |
| 地址编辑 | 新增/修改地址 |
| 证件列表 | 身份证件管理 |
| 证件编辑 | 新增/修改证件 |
| 设置 | 账号设置 |
| 用户协议 | 静态文本页 |
| 隐私政策 | 静态文本页 |
| 关于我们 | 静态文本页 |

**划分理由**：设置类页面访问频率最低，但页面数量多（尤其是协议、政策等静态页面会增加体积）。单独分包可确保这些页面的资源完全不影响启动性能。

---

## 预加载（preloadRule）策略

预加载的核心逻辑是**在用户停留当前页面时，后台静默下载"下一步最可能访问"的分包**，消除跨分包跳转的白屏等待。

| 当前页面 | 预加载分包 | 网络条件 | 理由 |
|----------|-----------|----------|------|
| 首页 | `product`（商品分包） | all | 用户浏览首页后，点击商品的概率极高 |
| 购物车 | `order`（订单分包） | all | 购物车的下一步操作通常是结算 |
| 个人中心 | `settings`（设置分包） | wifi | 个人中心聚集了所有设置入口，但这是低频场景，仅 Wi-Fi 下预加载节省流量 |
| 订单详情 | `aftersale`（售后分包） | wifi | 查看订单后可能发起退款/投诉，但属于小概率事件，仅 Wi-Fi 预加载 |

> `network: 'all'` 适用于高频跳转场景（移动网络也预加载）。
> `network: 'wifi'` 适用于低频或非必须场景（节省用户流量）。
> **不预加载**的分包：用户主动导航时微信会自动下载，无需提前占用带宽。

## 主包体积优化建议

除了分包策略外，以下措施可进一步压缩主包体积：

1. **图片资源上 CDN**：TabBar 图标以外的图片移至阿里云 OSS，主包只保留必需的本地图标。
2. **按需引入组件库**：NutUI 使用 Tree Shaking，仅引入主包页面实际用到的组件。
3. **共享组件审查**：`src/components/` 中的组件如果只在分包页面使用，可移至对应分包的 `components/` 目录（分包独享组件），避免被主包打包。
4. **字体文件**：避免在主包内嵌入自定义字体，改用网络字体或系统字体。
5. **检查 `assets/` 目录**：确认主包 `assets/` 下没有遗留的大图、视频或未压缩的图片。

---

## 分包体积限制总结

| 限制项 | 阈值 |
|--------|------|
| 主包大小 | < 2MB |
| 单个分包大小 | < 2MB |
| 所有分包总大小 | < 20MB（总上限） |
| 小程序总体积 | < 20MB |
